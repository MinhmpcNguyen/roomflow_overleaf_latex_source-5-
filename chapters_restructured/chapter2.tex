Chapter 1 defined two connected needs: editable furniture layout generation and layout-consistent visualization. This chapter turns those needs into system requirements. It first reviews existing AI-assisted interior design approaches, then identifies the practical gaps that the proposed system must address, and finally specifies the functional and non-functional requirements of RoomFlow.

\section{Survey of Existing Approaches}\label{sec:status-survey}

AI-assisted interior design has been explored through prompt-based image generation, LLM-based layout planning, hybrid layout-generation systems, and layout-conditioned image generation or editing. This section reviews these directions from the viewpoint of the proposed system requirements: natural-language input, spatially valid placement, editable layout state, layout-consistent rendering, and controlled image editing.

\subsection{Prompt-Based Interior Image Generation}\label{subsec:prompt-based-interior-image-generation}

Prompt-based interior image generation turns a user's natural-language description of a room into a visual concept. This approach is useful for early inspiration because it requires little effort and can quickly show possible styles, materials, lighting conditions, and atmospheres.

The main limitation is that the output is an image rather than an editable design state. A generated image may be visually attractive, but it does not provide stable object identities, dimensions, positions, orientations, or room-boundary constraints. As a result, the user cannot reliably move a sofa, preserve a door position, resize a wardrobe, or continue editing the scene as a real layout. The image model may also change the room shape, add or remove openings, hallucinate furniture, or ignore the requested arrangement. These issues make text-only image generation useful for inspiration but insufficient for layout design.

Therefore, prompt-based image generation is useful for inspiration but insufficient as the main design state. Early-stage interior design requires a representation that preserves object identities, dimensions, positions, and orientations so that the user can continue refining the room as an editable layout.

\subsection{LLM-Based Layout and Scene Planning}\label{subsec:llm-based-layout-and-scene-planning}

The second direction is LLM-based layout and scene planning. Systems such as LayoutGPT show that large language models can act as visual planners and generate structured scene descriptions from text \cite{al2023LayoutgptCompositionalVisual}. This direction is relevant because it allows users to express design requirements naturally, without manually writing object lists or spatial constraints.

However, direct layout generation by language models remains limited. A language model can suggest plausible furniture items and relations, but it does not automatically guarantee geometric validity. In indoor layout design, this weakness may appear as furniture placed outside the room, objects overlapping each other, blocked doors or windows, or arrangements that only work for simple rectangular rooms. These errors are especially problematic for non-professional users because they may not notice the issue until a later stage.

This direction suggests an important design principle for the thesis. Language models should be used for semantic planning, such as understanding user intent, identifying required objects, and proposing object relations. They should not be the only component responsible for final object coordinates.

\subsection{Hybrid AI Layout Generation}\label{subsec:hybrid-ai-layout-generation}

The third direction is hybrid AI layout generation. Systems such as I-Design, HOLODECK, and FlairGPT suggest that language models become more reliable when they extract objects, relations, or constraints before a separate placement or optimization stage handles geometry \cite{al2024IDesignPersonalized}, \cite{al2024HolodeckLanguageGuided}, \cite{Littlefair2025FlairgptRepurposingLlms}. This direction is closer to the goal of the thesis because it separates semantic understanding from physical placement.

The strength of this direction is that each module solves a more suitable subproblem. The language model handles flexible user input, while the placement or solver module checks spatial feasibility. This structure can reduce hallucinated coordinates and make the final layout easier to inspect.

The limitation is that many hybrid systems are designed mainly to demonstrate scene-generation algorithms or benchmark performance. They may generate a layout or a 3D scene, but they do not always provide the full application loop needed by non-professional users: drawing a room, editing a 2D layout, inspecting a 3D preview, generating a render from the selected layout, saving artifacts, and revising the result. The proposed system adopts the hybrid principle but implements it as an interactive workflow with editable layout state and user-facing correction.

\subsection{Layout-Conditioned Image Generation and Editing}\label{subsec:layout-conditioned-image-generation-and-editing-survey}

Layout-conditioned image generation and editing use the selected layout or another structural reference to guide the image model. Unlike text-only generation, this approach gives the model visual evidence about the room structure, furniture arrangement, openings, and camera view that should be preserved in the result.

Ctrl-Room shows a related direction by separating layout and appearance generation for controllable room generation \cite{al2025CtrlRoomControllable}. Conditional image-generation methods also suggest that spatial evidence can improve output control. For the thesis, the relevant insight is that visualization should be treated as a stage derived from the layout, not as an independent text-to-image result.

The limitation is that even layout-conditioned generation may still drift from the selected design. The image model can change object identity, alter object scale, move the camera, add new furniture, or modify windows and doors. This means that the system must not only send a visual condition to the image model, but also define preservation constraints and evaluate whether the output remains faithful to the layout.

\subsection{General Assessment and Requirement Identification}\label{subsec:general-assessment-and-requirement-identification}

The surveyed approaches show that no single direction fully satisfies the needs of early-stage interior design for non-professional users. Prompt-based image generation provides fast visual inspiration but lacks editability. LLM-based planning provides natural-language interaction but cannot guarantee valid geometry by itself. Hybrid layout-generation systems improve spatial reasoning but are often not organized as an end-to-end user workflow. Layout-conditioned rendering improves visualization control but can still drift from the selected design. These observations lead to the practical requirements used to define the proposed system.

\section{Functional Overview}\label{sec:functional-overview}

After identifying the main requirements in Section 2.1, this section presents the proposed system from the functional perspective. The purpose is to show how the application supports a complete early-stage interior design workflow, from account access and room creation to layout generation, layout editing, 3D preview, perspective rendering, and supporting data management. Detailed use case specifications are presented in Section 2.3.

The system functions are organized around the two needs defined in Chapter 1. For editable layout generation, the application allows users to create a room, define fixed openings, arrange furniture manually, and request automatic layout variants from the AI-assisted pipeline. The generated layout is kept as structured and editable design data, so users can revise the result instead of treating it as a fixed output. For layout-consistent visualization, the application converts the selected layout into a 3D preview, uses the preview and layout metadata to generate a perspective image, and supports localized image editing when the user wants to modify a specific visual element.

The use case model includes three human actors. A Guest can register or log in. A User is an authenticated actor who performs the main design workflow, including room creation, manual editing, automatic layout generation, 3D preview, render generation, image editing, and artifact management. An Admin maintains shared system data such as user accounts and furniture records. External AI services are not modeled as human actors because they do not initiate user goals. They are supporting components invoked by the backend and are described later in the sequence and architecture diagrams.

\subsection{Business Process for Layout Design and Visualization}\label{subsec:business-process-layout-design-visualization}

Figure~\ref{fig:2-4} presents the main workflow of the system. After logging in, the user starts a design session by creating a new room or loading a saved room. A room contains the basic information required for later processing, including the room type, boundary, doors, windows, and design brief.

The design flow then continues through two possible layout paths. In the manual path, the user directly arranges furniture in the 2D workspace. In the automatic path, the system uses the room data, design brief, and furniture inventory to generate editable layout variants. The user reviews these variants, selects one layout, and may continue adjusting it manually. This loop keeps the automatic result as a starting point while preserving user control over the final arrangement.

After the layout is accepted, the system converts it into a 3D preview. From the selected camera view, the user can generate a perspective render image. If the render needs further refinement, the user can either return to layout editing and render again, or request a localized image edit on the existing render. The workflow ends when the user saves the layout, render, or edited render for later reuse.

The revision loop in Figure~\ref{fig:2-4} highlights that rendering is not a separate one-way step: if the generated perspective image reveals a layout issue, the user can return to the editable layout before generating a new visualization.

Admin management is outside the main design flow. It maintains user accounts and furniture records so that the design workflow can operate with valid users and consistent inventory data.

\hline
NFR-04 & Visualization consistency & Rendered images should preserve the main room structure, camera view, openings, and major furniture arrangement from the selected layout. \\
\hline
NFR-05 & Reliability & The system should handle invalid room geometry, failed AI requests, missing furniture assets, and rendering failures without crashing the whole application. Long-running AI calls should expose progress and a clear failure message; retry should be possible when the provider or network fails. \\
\hline
NFR-06 & Maintainability & The system should separate frontend interaction, backend processing, solver logic, inventory data, persistence, and AI-provider integration. \\
\hline
NFR-07 & Security & Authentication data must be protected, raw tokens and passwords must not be logged, and administrator functions must be restricted to users with admin permission. \\
\hline
NFR-08 & Performance & Manual editing should feel immediate in the browser. As a design target, ordinary drag, rotate, and selection operations should update the 2D workspace within about 100 ms on typical laptop hardware. Long-running layout-generation and image-generation tasks may take longer, but the system should report progress within a few seconds and provide a clear retry path if the provider or network fails. \\
\hline
NFR-09 & Traceability & Generated layouts, renders, and edited renders should be linked to their source room, selected layout, prompt package, and relevant metadata. \\
\hline
NFR-10 & Cost control & Since AI generation may use external services, the system should avoid unnecessary repeated calls and reuse stored artifacts when possible. \\
\hline
\end{xltabular}
\endgroup

Usability and editability are the most important user-facing requirements. The target users are not professional designers, so they must be able to create a room, request a layout, adjust the result, and generate a render through direct interaction.

Spatial consistency and visualization consistency are the most important technical requirements. The system should not only produce visually pleasing outputs. It must preserve the design structure that the user created or selected. This requirement connects directly to the two problems introduced in Chapter 1.

Reliability, maintainability, traceability, and cost control are necessary because the system uses multiple AI and geometry-processing stages. Saving intermediate artifacts and separating modules make the system easier to debug, evaluate, and extend.
