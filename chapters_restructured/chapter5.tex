\chapter{Photorealistic Interior Image Generation and Editing}

This chapter presents the Layout-Guided Photorealistic Rendering and Editing component of RoomFlow. While Chapter 4 focuses on generating a structured and editable furniture layout, this chapter explains how the selected 3D layout is converted into a photorealistic interior design output. The main goal is not to generate an interior image freely from text, but to produce a realistic render that remains consistent with the room geometry, camera view, openings, and furniture arrangement defined by the selected layout. To achieve this, RoomFlow uses the 3D preview as a layout reference, captures a snapshot from the chosen camera view, and combines it with structured preservation constraints and rendering instructions. The chapter also describes the image-editing workflow, where users can modify selected visual details while preserving the main room structure and unrelated furniture.

\section{Overall Processing Flow}\label{sec:problem2-overall-processing-flow}

\hyperref[fig:5-1]{Figure~5.1} summarizes how RoomFlow turns a selected layout into visual render artifacts. The editable layout produced in Chapter 4 acts as the structural source of truth for this pipeline. It contains the room geometry, openings, furniture objects, positions, orientations, and display metadata. The image-generation and image-editing workflows do not replace this structured layout. Instead, they create visual artifacts derived from it, so that the user can inspect the design in a more realistic form while the underlying layout remains editable.

The flow has two connected branches. The first branch is photorealistic generation. It converts the selected layout into a 3D preview, captures a snapshot from the chosen camera view, and compiles a generation package for the image model. This package combines the snapshot, visible-object information, visual presets, and preservation constraints. By using the layout-derived snapshot as visual evidence, the system reduces the risk of generating an attractive image that no longer matches the selected room structure and furniture arrangement.

The second branch is localized editing. It starts from an existing render and applies a user-requested change to a target object or region. The edit request may involve adding, removing, replacing, or recoloring an element. Before sending the request to the image model, RoomFlow prepares the source render, edit instruction, target scope, and preservation constraints. The purpose is to apply the intended change while keeping unrelated furniture, camera framing, room shell, and openings as stable as possible.

Both branches produce traceable render artifacts. A generated render remains linked to the selected layout, while an edited render remains linked to the source render and, when possible, to the original layout. This traceability is important because the image is only a visual output, not the authoritative design state. If a generated or edited image changes the room geometry, opening positions, camera view, or major furniture arrangement unintentionally, it is treated as a rendering or editing failure rather than a valid update to the layout.

The following sections describe these two branches in detail. Section 5.2 presents the image-rendering workflow, including snapshot-based layout conditioning, scene and visible-object contracts, and structured prompt packaging. Section 5.3 presents the image-editing workflow, including edit request construction, target scope, target validation, preservation constraints, and edited render output.

\section{Layout-Guided Photorealistic Rendering}\label{sec:problem2-image-generation}

The rendering branch converts the selected editable layout into a photorealistic perspective render. The input is not a text prompt alone, but a rendering package derived from the selected 3D layout. This package contains three main groups of information: a snapshot captured from the selected camera view, visible-object and preservation information, and visual presets for the desired appearance. The result is an image that corresponds to the chosen layout, while the original layout remains the main version that users can continue to edit.

For example, consider a living-room layout viewed from the selected camera position. The snapshot shows the room from that view. The visible-object information records the main objects seen by the camera, such as the sofa, coffee table, TV console, shelving, rug, and window. The preservation information specifies what should not change, such as the room shell, window position, camera framing, and major furniture arrangement. The visual presets describe the intended appearance, such as warm materials with soft daylight.

\hyperref[fig:5-2]{Figure~5.2} summarizes this layout-guided rendering flow. The layout-derived input contains the 3D snapshot, visible-object map, and visual preset. The conditioning compiler then builds the visual guide, scene contract, object contract, and prompt package. During provider execution, the prompt package is sent to the image model, the output and metadata are saved, and the perspective image is returned as a photorealistic render linked to the selected layout. The following subsections explain these steps in order: snapshot-based layout conditioning, visible-object and preservation contracts, and visual preset control with rendering request assembly.

\subsection{Snapshot-Based Layout Conditioning}\label{subsec:problem2-technique-snapshot}

Snapshot-based layout conditioning is the first step used to keep the generated render close to the selected layout. A snapshot is a captured view of the 3D preview from the camera angle chosen by the user. It shows the visible room structure, openings, furniture positions, and approximate object shapes before photorealistic rendering. In RoomFlow, this snapshot acts as layout evidence for the image model, so the model does not have to infer the room arrangement from text alone.

\begin{figure}[!htbp]
\centering
\includegraphics[width=0.94\textwidth]{figures/chapter5/snapshot_layout_conditioning_example.png}
\caption{Snapshot-based layout conditioning from a real living-room case. The image captures the selected 3D camera view before photorealistic rendering.}
\label{fig:5-snapshot-layout-conditioning}
\end{figure}

The main purpose of the snapshot is to reduce ambiguity during image generation. If the image model only receives a text instruction such as ``render this bedroom in a Japandi style,'' it may create a visually pleasing room but change the camera view, move furniture, add extra objects, or alter the window position. By providing the selected 3D view, RoomFlow gives the model a concrete reference for the room shell, camera framing, furniture grouping, and spatial arrangement that should be preserved.

The snapshot is not expected to be a final realistic image. It may contain simple geometry, basic materials, and approximate asset shapes. Its role is to communicate layout structure rather than visual quality. The image model is then expected to improve realism, lighting, materials, and atmosphere while following the spatial evidence provided by the snapshot.

This design also keeps the generated render connected to the editable layout. The render is considered a visual output derived from the selected layout, not a replacement for it. If the resulting image changes the room shape, camera framing, opening positions, or major furniture arrangement, the change is treated as layout drift rather than a valid improvement. This issue is evaluated explicitly in Chapter 8.

\subsection{Visible-Object and Preservation Contracts}\label{subsec:problem2-technique-contract}

Visible-object and preservation contracts specify which parts of the selected view should remain stable during rendering. The snapshot shows the layout visually, but it may not clearly state which objects are important or which changes are unacceptable. RoomFlow therefore adds preservation information to make the protected layout elements explicit.

The contract combines three types of evidence. A camera-aligned structural guide highlights visible object boundaries and room surfaces in the same view as the snapshot. Visible-object information records the main objects seen by the camera, including their categories, approximate image regions, and layout roles. Preservation constraints state what should not change, such as camera framing, room shell, door and window positions, object count, object scale, relative depth order, and major furniture arrangement.

Instead, the contracts reduce ambiguity: the model may improve materials, lighting, shadows, and atmosphere, but the room structure and visible furniture arrangement should remain tied to the selected editable layout. \subsection{Visual Preset Control and Rendering Request Assembly}\label{subsec:problem2-prompt-optimization}

The final rendering request combines layout evidence with visual preset control. The provider-facing request is organized into stable sections: source snapshot, visible-object information, preservation requirements, style preset, lighting preset, scenery preset, and negative constraints. This structure makes requests easier to inspect, compare, and reproduce across experiments.

The style, lighting, and scenery presets are separated from geometry. A style preset may request modern, minimalist, boho, or warm material choices. A lighting preset may request daylight, soft evening light, or studio-like clarity. A scenery preset may describe the outside view or environmental mood. These presets should improve visual appearance without changing the underlying layout goal, as illustrated by the generated render in \hyperref[fig:5-3]{Figure~\ref*{fig:5-3}} from the same living-room case used in the snapshot and preservation examples.

\section{Localized Image Editing}\label{sec:problem2-image-editing}

Localized image editing allows the user to revise a generated render without restarting the complete layout and rendering workflow. Instead of generating a new image from a blank prompt, the system starts from an approved source render and applies a specific edit operation, such as adding, removing, replacing, or recoloring an element. The purpose is to make a targeted visual change while keeping the camera view, room shell, openings, and unrelated furniture stable.

\hyperref[fig:5-4]{Figure~\ref*{fig:5-4}} summarizes the localized editing flow. The source and target input consists of the approved source render, the user's edit instruction, and the target scope. The edit request compiler then performs target validation, builds an edit guide, writes a preservation contract, and creates the provider request. During provider execution, the image model produces an edited image, the output and metadata are saved, and the final artifact remains linked to the source render and target scope so that later evaluation can check both edit success and unintended changes outside the edited region.

\centering

\caption{Localized image-editing flow from source-and-target input through edit request compiler and provider execution.}

This editing branch is useful because not every visual change requires a new layout or a new full render. For example, the user may want to add a wall artwork, recolor a sofa, remove a small object, or replace one furniture item while keeping the rest of the room unchanged. These operations are treated as image-level refinements. They do not update the editable 2D layout automatically, because the structured layout remains the main design state of RoomFlow.

\subsection{Edit Request and Target Scope}\label{subsec:problem2-edit-target-scope}

Each localized edit is defined by two elements: an edit request and a target scope. The edit request specifies what the user wants to change, such as adding, removing, replacing, or recoloring an element. The target scope specifies where the change should happen in the source render.

When the target corresponds to a known layout object, RoomFlow can reuse its object identity and approximate visible region. When the target exists only in the generated render or cannot be reliably linked to layout data, the user can mark a region directly on the image. This allows the same editing workflow to support both layout-linked edits and image-only refinements.

The target scope reduces ambiguity because the image model is told both what to change and where the change should occur. For example, adding wall art is more controlled when the target is a selected wall region, and recoloring a sofa is safer when the target is linked to the visible sofa area. This localized formulation helps apply the requested edit while limiting unnecessary changes to the camera view, room shell, openings, and unrelated furniture.

\subsection{Target Validation and Preservation Constraints}\label{subsec:problem2-edit-preflight}

Before sending the edit request to the image model, RoomFlow checks whether the selected target can be edited clearly. If the target is linked to a known layout object, the system uses its visible region as the editing area. If the target is selected directly on the image, the marked region is used instead. This validation step helps ensure that the requested change has a clear location in the source render.

The system then defines preservation constraints for the parts of the image that should remain unchanged. These constraints protect the camera view, room shell, openings, major furniture arrangement, and unrelated objects. For example, recoloring a sofa should not change the wall, replacing a chair should not move the table, and removing a small object should not remove nearby furniture.

\hyperref[fig:5-5]{Figure~\ref*{fig:5-5}} illustrates this process with a real edit-flow example. The selected target region shows where the requested operation is allowed, while the rest of the room is treated as protected context. This step does not guarantee perfect editing, but it reduces ambiguity and provides a clearer basis for evaluating edit success and structure preservation in Chapter 8.

\subsection{Edited Render Output}\label{subsec:problem2-edited-render-output}

The output of localized image editing is an edited image that is connected to the original approved render and the selected editing area. The expected result depends on the requested operation. A recolor operation should change only the selected object. A remove operation should remove the target element while repairing the surrounding visual region. A replacement operation should change the intended item while preserving its approximate location, scale, and relation to nearby furniture. An add operation should insert the requested element into the target region without changing unrelated parts of the room.

\hyperref[fig:5-6]{Figure~\ref*{fig:5-6}} shows the final edited render artifact from a real localized add operation. The edited image is stored together with its source render and target region so that the result can be inspected later. This link is important because evaluation must check both whether the requested edit was applied and whether the rest of the room was preserved.

The edited render remains a visual artifact, not the structural source of truth. The editable layout remains the main design state of RoomFlow. If an image-level edit changes the camera framing, room geometry, opening positions, or major furniture arrangement, the change is treated as unintended drift rather than a valid layout update. The current system can save and evaluate edited renders, but it does not automatically synchronize image-only edits back into the editable 2D layout.
