\chapter{Experimental Evaluation}

This chapter evaluates the two problems defined in Chapter 1. Section 7.1 evaluates Problem 1: automatic editable layout generation. Section 7.2 evaluates Problem 2: snapshot-guided image generation and controlled image editing.

\section{Layout Generation Evaluation}\label{sec:eval-layout-generation}

The goal of this section is to check whether RoomFlow can generate valid, request-aware, editable, and stable furniture layouts. The evaluation is organized by research questions rather than by individual cases. Individual cases are used later as benchmark scenarios to explain the aggregate results.

\subsection{Evaluation Objectives}\label{subsec:eval-layout-objectives}

The layout evaluation answers three research questions:

\begin{itemize}
\item \textbf{RQ1:} Does the full system perform better than external AI layout baselines such as I-Design and FlairGPT?
\item \textbf{RQ2:} Do the internal components of the full system contribute to layout quality?
\item \textbf{RQ3:} Does the system remain robust across repeated runs and increasingly difficult room geometries, including non-rectangular rooms with fixed openings?
\end{itemize}

\subsection{Evaluation Protocol}\label{subsec:eval-layout-protocol}

The layout benchmark is executed as a controlled offline evaluation. Each case contains a room type, a boundary polygon, optional doors and windows, and a natural-language design brief. One \emph{case-run observation} is one system output for one benchmark case in one repeated run. Each benchmark case is executed twice to reflect the non-deterministic behavior of AI-assisted generation, and scores are then aggregated by scenario group and opening condition. This setup separates case diversity from repeated-run variation.

The full RoomFlow pipeline, the external baselines, and the ablation variants receive equivalent case information after conversion into their expected input formats. A missing output, failed run, or non-convertible result is scored as zero because it does not provide a usable layout for the user. The tables report subgroup means and, where repeated observations are summarized in one cell, mean \(\pm\) standard deviation. Paired deltas are computed on matched case-run observations; confidence intervals are reported for the mean paired difference, and p-values are adjusted with Holm correction when multiple comparisons are shown together.

The external baseline comparison is therefore not a comparison of native scores reported by those systems. It is a task-level comparison after adapter conversion. This makes the results easier to interpret for the thesis benchmark, but it also means that baseline performance may be affected by prompt conversion and output parsing quality.

\subsection{Compared Systems}\label{subsec:eval-layout-compared-systems}

The evaluation compares the full RoomFlow pipeline against external baselines and internal ablations. For fairness, each baseline output is converted to the same common 2D schema before scoring. The conversion keeps the object categories, approximate footprints, positions, and orientations that can be recovered from the baseline output. The same geometric checks and scoring scripts are then applied to all systems. This normalization is necessary because the compared systems expose different output formats, but it is also a threat to validity discussed in Section~\ref{sec:eval-threats-to-validity}.

\subsection{Evaluation Metrics}\label{subsec:eval-layout-metrics}

Before scoring the full benchmark, each output is normalized into the same 2D layout schema. All four primary layout metrics are reported on a 0--100 scale, where higher is better. The overall score is a weighted combination that prioritizes physical usability while still rewarding prompt compliance and usable object coverage. The evaluation first asks whether the layout is physically valid, then whether it follows the prompt, then whether it is usable as a room, and finally whether the result is stable across repeated runs.

Table~\ref{tab:7-4} explains how each metric group should be interpreted before the numerical results are presented.

\hline
Does the layout follow the prompt? & Request following & Checks whether the requested objects, counts, and key relations are present. For example, a sofa should face the TV if the prompt asks for a media area. \\

The four component metrics are designed specifically for this benchmark, rather than copied directly from a single standard benchmark. They are informed by evaluation signals commonly used in prior indoor-layout and text-to-3D scene-generation studies, including out-of-boundary, overlap, accessibility, object-selection, and instruction-alignment checks~\cite{al2023LayoutgptCompositionalVisual,al2024IDesignPersonalized,al2024HolodeckLanguageGuided,Littlefair2025FlairgptRepurposingLlms}. All component scores are reported on a 0--100 scale, using \(\mathrm{clip}_{0,100}(x)=\min(100,\max(0,x))\). Let \(n_{\mathrm{oob}}\), \(n_{\mathrm{overlap}}\), \(n_{\mathrm{open}}\), and \(n_{\mathrm{access}}\) denote the numbers of out-of-room blocking objects, overlapping blocking-object pairs, opening-clearance violations, and blocked door-to-room-center access paths. Let \(A_{\mathrm{oob}}\) and \(A_{\mathrm{overlap}}\) denote the corresponding total violation areas in \(\mathrm{mm}^{2}\).

\noindent\textbf{Geometry validity.}
This metric captures basic physical feasibility, following the use of out-of-boundary and overlap-related checks in LayoutGPT, I-Design, and FlairGPT~\cite{al2023LayoutgptCompositionalVisual,al2024IDesignPersonalized,Littlefair2025FlairgptRepurposingLlms}.
\[
\begin{aligned}
S_{\mathrm{geometry}}=\mathrm{clip}_{0,100}\bigl(
100
&-22n_{\mathrm{oob}}-24n_{\mathrm{overlap}}
-16n_{\mathrm{open}}-12n_{\mathrm{access}}\\
&-\min(A_{\mathrm{oob}}/40000,18)
-\min(A_{\mathrm{overlap}}/50000,18)
\bigr).
\end{aligned}
\]
The area terms make large boundary or overlap violations worse than small numerical contacts, while the clipping keeps the score within the common 0--100 range.

\noindent\textbf{Request following.}
This metric checks whether the generated layout satisfies the case-specific prompt requirements encoded in the benchmark, including required objects, expected object counts, and key spatial relations; it follows the broader instruction-alignment perspective used in LLM-based scene-generation systems~\cite{al2023LayoutgptCompositionalVisual,al2024IDesignPersonalized}.
\[
S_{\mathrm{request}} =
\begin{cases}
100, & N_{\mathrm{checks}}=0,\\
100 \cdot \dfrac{N_{\mathrm{passed}}}{N_{\mathrm{checks}}}, & N_{\mathrm{checks}}>0.
\end{cases}
\]
Here \(N_{\mathrm{checks}}\) is the number of benchmark checks applicable to the case and \(N_{\mathrm{passed}}\) is the number satisfied by the output.

\noindent\textbf{Arrangement quality.}
This metric is a conservative proxy for practical room usability, motivated by layout-coherence, accessibility, and functional-placement criteria discussed in HOLODECK and FlairGPT~\cite{al2024HolodeckLanguageGuided,Littlefair2025FlairgptRepurposingLlms}.
\[
S_{\mathrm{arrangement}}=\mathrm{clip}_{0,100}\bigl(
100
-18n_{\mathrm{overlap}}-14n_{\mathrm{oob}}
-12n_{\mathrm{open}}-10n_{\mathrm{access}}
\bigr).
\]
Unlike geometry validity, this score uses only count-based penalties because it is meant to summarize whether repeated spatial incidents make the room hard to use.

\noindent\textbf{Valid object completeness.}
This metric is adapted from object-selection and object-coverage signals in prior work, but applies a stricter definition by counting only main blocking furniture that remains usable after invalid out-of-room or overlapping objects are excluded~\cite{al2024IDesignPersonalized,al2024HolodeckLanguageGuided,Littlefair2025FlairgptRepurposingLlms}.
\[
S_{\mathrm{valid\_count}} =
\begin{cases}
100, & N_{\min}\ \text{is not specified},\\
\mathrm{clip}_{0,100}\bigl(100-18\max(N_{\min}-N_{\mathrm{valid}},0)\bigr), & \text{otherwise}.
\end{cases}
\]
Here \(N_{\min}\) is the minimum expected number of usable blocking objects and \(N_{\mathrm{valid}}\) is the number that remains after excluding invalid objects.

The final score combines the four components as:
\[
\begin{aligned}
S_{\mathrm{overall}}=\mathrm{clip}_{0,100}\bigl(
&0.30S_{\mathrm{geometry}}
+0.25S_{\mathrm{request}}\\
&+0.25S_{\mathrm{arrangement}}
+0.20S_{\mathrm{valid\_count}}
\bigr).
\end{aligned}
\]
The weights give the largest share to geometry because a layout that crosses room boundaries or blocks openings is not usable even if it lists the right objects. Request following and arrangement receive equal weight because an early design must both answer the brief and remain functional. Valid object completeness receives a lower weight because it is a completeness check, not a full usability judgment. Failed or missing outputs are counted as zero because they do not produce a usable auto-fill layout.

\subsection{Scenario-Level Analysis}\label{subsec:eval-layout-scenario-analysis}

To make the metric interpretation concrete, this subsection shows three representative cases before the aggregate result tables. The snippets are shortened normalized outputs rather than complete system artifacts. They keep only the fields needed to explain the score: room description, main object list, component scores, and violation count. Full artifacts contain additional geometry, metadata, and diagnostics.

These examples also clarify why the metric groups are separated. A system can name the requested furniture and therefore look acceptable on request following, but still fail spatial validity if the objects block a door, overlap, or cross the room boundary. A system can also keep enough valid objects but arrange them in a way that is hard to use. The metric set therefore reads each layout from four angles: what the user asked for, whether the objects physically fit, whether the room is usable, and whether enough usable furniture remains.

\subsection{Comparison with External Baselines}\label{subsec:eval-layout-external-baselines}

The external comparison contains 100 case-run observations per system: 40 simple sanity observations, 40 regular rectangular observations, and 20 non-rectangular observations. A case-run observation means one execution of one benchmark case by one system. Each benchmark case is executed twice to reflect the non-deterministic behavior of AI-assisted generation. Therefore, the \textit{Overall} column in the subgroup tables is reported as mean \(\pm\) standard deviation over case-run observations. The component metrics remain as means to keep the tables readable. Failed outputs are assigned zero. The subgroup means and deviations are descriptive; the pooled paired confidence intervals reported afterward provide the inferential summary.

\subsection{Discussion}\label{subsec:eval-layout-discussion}

The layout evaluation supports three conclusions. First, RoomFlow full is stronger than the external baselines on overall layout quality, mainly because it improves geometry and arrangement validity. Second, ablation results show that semantic planning, solver validation, and output selection each contribute different types of robustness. Third, the advantage becomes clearer in difficult rooms, where non-rectangular boundaries and openings expose the limitations of direct or weakly constrained layout generation.

\subsection{Compared Workflows}\label{subsec:eval-image-compared-workflows}

The evaluation contains two separate comparisons. Six variants belong to image generation. Two other variants belong to image editing and are not counted as generation variants.

The image-flow evaluation compares six provider-facing generation workflows. The abbreviations used in tables and charts are defined here: SP is \emph{simple prompt}; CO is \emph{object contract}; CP is \emph{contract plus visual preflight}; LG is \emph{locked geometry}; LVP is \emph{LLM visual preflight}; and SE-ESG is the proposed \emph{single-edit edge/silhouette-guided} workflow. The two editing workflows are reported separately and are not part of this six-variant generation comparison.

\medskip
\noindent\textbf{Image-generation variants.}\par\smallskip

All six generation variants start from the same selected layout, camera view, 3D snapshot, and visual preset. They differ only in the guidance supplied before the image model produces the render.

\noindent\textbf{Textual guidance.} An \emph{object contract} is a structured text list of visible objects, openings, and preservation requirements. A \emph{visual preflight} is a short inspection step in which an LLM examines the source snapshot and metadata, identifies likely risks, and writes concise locks such as ``keep the window visible'' or ``do not merge the foreground furniture.'' The preflight produces text instructions; it does not modify the image. The prompt policy then determines which of these descriptions and locks are sent to each variant.

\noindent\textbf{Visual guidance.} The common 3D snapshot provides the camera, room shell, openings, and furniture. The locked geometry atlas adds crops of important walls and object shapes. The edge/silhouette guide explicitly marks boundaries that should remain traceable after photorealization. Figure~\ref{fig:image-create-guidance-inputs} shows only these image-based inputs using one actual benchmark case.

\subsection{Evaluation Metrics}\label{subsec:eval-image-metrics}

The image-generation metrics are reported on a 0--1 scale, where higher means that the generated render remains more similar to the source layout on the measured property. Edge structure asks whether important wall and furniture outlines remain traceable. Depth structure asks whether near/far organization and occlusion remain similar. DINO patch similarity compares learned visual features in corresponding local regions, while DINO object similarity compares features inside the same visible-object boxes. The macro score is the unweighted mean of these four complementary preservation metrics; it is a balanced structural indicator rather than a complete judgment of visual quality.

Figure~\ref{fig:image-metric-explanations} gives a visual reading of the four metrics on one real source-render pair. Edge structure compares wall and furniture contours. Depth structure checks whether near and far regions, occlusions, and depth boundaries remain in the same organization. DINO patch similarity compares learned features at corresponding local image patches. DINO object similarity crops the same visible-object boxes from both images and compares their learned features. The grids, arrows, and boxes in the figure are explanatory overlays; the evaluator computes edge masks, learned depth maps, and DINO features internally.

\begin{figure}[!htbp]
\centering
\includegraphics[width=0.96\textwidth]{figures/chapter4/image_flow_cases/image_metric_explanations.png}
\caption{Visual interpretation of the four layout-preservation metrics.}
\label{fig:image-metric-explanations}
\end{figure}

The edit metrics combine manual and automated checks. The human edit score uses a fixed operation-specific rubric: the requested add, remove, replace, or recolor must be applied; the intended target or region must be respected; the camera and room shell must remain stable; unrelated objects should be preserved; and obvious artifacts should be avoided. Each criterion is scored as passed or failed, and the score ratio is the number of passed criteria divided by the number applicable to that operation. These are single-rater thesis annotations rather than a blinded multi-rater user study. Edge and depth metrics measure structural preservation, but they are interpreted together with edit success because an under-edited image may look structurally close while failing the user's request.

\subsection{Discussion}\label{subsec:eval-image-discussion}

The image evaluation supports three conclusions. First, snapshot-guided rendering is better treated as a structured workflow than as a plain text prompt. SE-ESG gives the best macro trade-off among the tested create variants. Second, preset fidelity is sufficiently high in the completed checks to support using style, lighting, and scenery presets as controllable user-facing options. Third, object-scope preflight improves localized edit success, especially for recolor, although it costs more tokens and can reduce edge/depth similarity when the edit is more decisive.

\section{Threats to Validity}\label{sec:eval-threats-to-validity}

\noindent\textbf{Internal validity.}
Implementation differences may affect the comparison with external baselines because their prompts and outputs must be converted to the common benchmark schema. The adapters preserve recoverable object categories, footprints, positions, and orientations, and all systems are scored by the same scripts, but conversion quality may still influence the result.

\noindent\textbf{External validity.}
The benchmark focuses on controlled living-room and bedroom cases. It does not cover kitchens, bathrooms, offices, construction documents, or the full diversity of real apartments and user preferences.

\noindent\textbf{Construct validity.}
Geometry, request following, arrangement, image-fidelity, and edit metrics capture important properties, but they cannot fully represent subjective design preference or professional interior-design quality. The human edit scores are also single-rater annotations rather than a formal user study.

\noindent\textbf{Conclusion validity.}
The number of cases and two repeated runs limit statistical power, especially for provider-based image generation. Confidence intervals and Holm adjustment reduce overstatement, but the findings remain evidence for the tested setting rather than proof that the workflow solves every interior-design task.
